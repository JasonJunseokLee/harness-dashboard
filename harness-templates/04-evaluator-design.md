# 04. 평가 에이전트 설계 — 쇼핑몰 데이터 수집 크롬 익스텐션 맞춤판

> 원본 문서의 핵심 원칙을 유지하되, Chrome Extension + 비개발자 사용자 + 범용 DOM 파서 컨텍스트에 맞게 구체화

---

## 왜 이 프로젝트에서 Evaluator가 특히 중요한가

- **쇼핑몰마다 DOM 구조가 다르다** → Generator가 "추출 잘 됨"이라고 판단해도 실제 네이버쇼핑·쿠팡·11번가에서 깨질 수 있다
- **비개발자 최종 사용자** → UX 실수(버튼 위치, 오류 메시지 부재)가 즉각 이탈로 이어짐
- **외부 API 전송** → Blast Radius가 높다. 잘못된 데이터가 고객사 플랫폼으로 나가면 되돌리기 어렵다

---

## 채점 기준 (이 프로젝트 전용)

### 1. 추출 정확도 (Extraction Accuracy) — 임계값: 9/10
- 상품명·가격·이미지 URL이 실제 페이지와 일치하는가?
- 옵션(색상, 사이즈 등)이 누락 없이 파싱되는가?
- 가격이 정가·할인가·배송비를 구분하여 추출하는가?
- 리뷰 데이터가 최신순으로 정렬·수집되는가?

### 2. 범용성 (Cross-site Compatibility) — 임계값: 8/10
- 최소 3개 이상 쇼핑몰(네이버쇼핑·쿠팡·11번가 기준)에서 동작하는가?
- 특정 쇼핑몰 셀렉터에 하드코딩된 로직이 없는가?
- DOM 구조가 바뀌었을 때 에러 없이 최선 결과를 반환하는가?

### 3. 전송 안전성 (Transmission Safety) — 임계값: 9/10
- 인증 토큰이 노출되지 않고 안전하게 전달되는가?
- API 응답 실패 시 사용자에게 명확한 피드백이 표시되는가?
- 중복 전송이 방지되는가?

### 4. 팝업 UX (Popup Usability) — 임계값: 8/10
- 비개발자가 가이드 없이 클릭 한 번으로 수집·전송할 수 있는가?
- 추출 미리보기에서 실제 수집 데이터가 검증 가능한가?
- 수집 상태(성공/실패/진행중)가 명확히 표시되는가?

### 5. 코드 품질 (Code Quality) — 임계값: 7/10
- Chrome MV3 규격을 벗어난 API 사용이 없는가?
- Content Script ↔ Background ↔ Popup 메시지 패싱이 타입세이프한가?
- Chrome Storage 사용이 sync/local 용도에 맞게 분리되어 있는가?

## 합리화 방지 — 이 프로젝트 전용 예시

**추출 정확도:**
```
❌ "가격 파싱이 대부분의 경우에 작동합니다"
✅ "쿠팡 옵션 선택 전 가격은 추출되나, 옵션 선택 후 동적 변경 가격은
    미추출. 추출 정확도 -2점. DOMObserver 적용 필요."
```

**팝업 UX:**
```
❌ "사용자가 설정을 통해 해결할 수 있습니다"
✅ "엔드포인트 미등록 상태에서 수집 클릭 시 에러 메시지 없음.
    비개발자 사용자는 익스텐션이 고장났다고 판단할 것. UX 점수 -3점."
```

**전송 안전성:**
```
❌ "토큰은 로컬 스토리지에 있어 안전합니다"
✅ "인증 토큰이 content script에서 직접 접근 가능 → XSS 위험.
    background service worker를 통한 토큰 중개로 수정 필요."
```

## Blast Radius 분류 (이 프로젝트 기준)

| 분류 | 예시 | 승인 수준 |
|------|------|----------|
| **Local** | DOM 파서 로직 수정, 팝업 UI 변경, 단위 테스트 | 자동 진행 |
| **Shared** | Chrome Storage 스키마 변경, manifest.json 권한 추가 | 경고 후 진행 |
| **External** | 실제 외부 API 엔드포인트 호출, 고객사 서버 전송, 크롬 웹스토어 배포 | **인간 승인 필수** |

> 특히 `host_permissions` 확장은 스토어 심사 재검토 트리거. 변경 전 반드시 확인.

## Quality Loop — 스프린트별 적용

```
1. 구현 — Content Script / Background / Popup 분리 구현
2. 리뷰 — 각 모듈의 메시지 패싱 인터페이스 재확인
3. 평가 — 아래 Evidence Gate 통과 여부 확인
4. 수정 — 실패 쇼핑몰 커버 / UX 개선
5. 줌아웃 — 다른 쇼핑몰 페이지에서 회귀 없음 확인
6. 반복 — Evidence Gate 통과까지
7. 보고 — 테스트 쇼핑몰 목록 + 추출 성공률 수치로 보고
```

### Evidence Gate (이 프로젝트 전용)

| 항목 | 필요한 증거 |
|------|-----------|
| 추출 정확도 | 네이버쇼핑·쿠팡·11번가 각 3개 상품 URL + 추출 결과 JSON |
| 범용성 | 하드코딩 셀렉터 없음 (grep 결과) |
| 전송 안전성 | Network 탭 스크린샷 (토큰 헤더 확인) |
| 팝업 UX | 비개발자 테스터 1인 클릭 완료 확인 |
| MV3 호환성 | `chrome.scripting` / `service_worker` 방식 사용 코드 예시 |
| 회귀 없음 | Vitest + Playwright E2E 통과 로그 |

## 멀티 페르소나 QA (Sprint 4+ 적용 권장)

Sprint 4(자동 전송·설정 UI 완성) 이후 완성품 수준 검수 시 적용.

| 페르소나 | 핵심 질문 | 이 프로젝트 검수 초점 |
|---------|---------|---------------------|
| **비개발자 운영자** | "코딩 없이 수집부터 전송까지 할 수 있는가?" | 팝업 UX, 오류 메시지 명확성, 엔드포인트 설정 난이도 |
| **개발자 (통합 담당)** | "API 응답을 신뢰할 수 있는가?" | 전송 포맷 일관성, 인증 구조, 에러 코드 명확성 |
| **데이터 품질 관리자** | "수집 데이터가 정확하고 완전한가?" | 필드 누락률, 인코딩 오류, 동적 가격 처리 |

## 실전 체크리스트

### 공통 (전 스프린트)
- [ ] 채점 기준이 위 5개 항목으로 적용되었는가?
- [ ] 각 기준에 구체적 임계값이 설정되었는가?
- [ ] 합리화 방지 지시가 프롬프트에 포함되었는가?
- [ ] Evaluator가 실제 쇼핑몰 URL에서 직접 테스트하는가?
- [ ] Blast Radius 분류에 따른 승인 수준이 지켜지는가?

### DOM 파서 관련 (Sprint 1~2)
- [ ] 테스트 쇼핑몰 3개 이상에서 추출 검증했는가?
- [ ] 동적 렌더링 페이지(SPA) 대응 여부를 확인했는가?
- [ ] 추출 실패 시 부분 결과 반환 vs 전체 실패 정책이 정해졌는가?

### 전송·인증 관련 (Sprint 3~4)
- [ ] 외부 API 호출 전 인간 승인 프로세스가 있는가?
- [ ] 토큰이 background service worker 외부로 노출되지 않는가?
- [ ] 전송 실패 재시도 로직의 무한루프 가능성을 확인했는가?

### Level 2 추가 (Sprint 5~6, 멀티 페르소나 QA)
- [ ] 3개 에이전트가 별도 컨텍스트에서 병렬 실행되는가?
- [ ] 비개발자 운영자 페르소나가 실제 타겟 사용자 행동을 모사하는가?
- [ ] 확정 결함 0개 기준으로 PASS/FAIL 판정하는가?
