# 11. 시각 QA 프로토콜 — 쇼핑몰 데이터 수집 크롬 익스텐션 맞춤판

> 코드 레벨 QA는 시각 일관성을 보장하지 못한다.
> 특히 Chrome Extension 팝업은 **픽셀 단위 공간 제약**이 있으며,
> 실제 쇼핑몰 페이지 위에 떠 있는 상태에서만 재현되는 버그가 존재한다.

---

## 핵심 교훈: 익스텐션 팝업은 일반 웹과 다르다

```
팝업은 width 360~400px, height 최대 600px의 극도로 좁은 공간이다.
빌드가 성공해도 팝업 안에서 레이아웃이 깨질 수 있다.

특히:
- 추출된 데이터가 많을 때 스크롤이 없으면 잘린다
- 로딩 스피너와 실제 데이터 영역의 높이가 달라 점프가 발생한다
- 배경 쇼핑몰 페이지와 팝업 색상이 충돌할 수 있다
```

**결론**: 반드시 실제 쇼핑몰 탭에 익스텐션을 띄운 상태에서 브라우저 시각 검증을 수행해야 한다.

---

## 1. 디자인 가이드 필수 (`.harness/design-system/design-guide.md`)

### 이 프로젝트에서 디자인 가이드가 담아야 할 항목

```markdown
## design-guide.md 필수 항목 (크롬 익스텐션 팝업 특화)

### 1. 팝업 캔버스 스펙
- [ ] 팝업 고정 너비: 360px or 400px 중 하나로 확정
- [ ] 최소/최대 높이: 컨텐츠별 범위 (e.g. 로딩: 200px / 데이터 풀: 최대 580px)
- [ ] 스크롤 전략: 전체 팝업 스크롤 vs 섹션별 내부 스크롤

### 2. 색상 토큰
- [ ] Primary (수집 시작/전송 버튼): 색상 + hover + disabled
- [ ] Danger (오류/전송 실패): 색상 + 배경
- [ ] Success (전송 완료): 색상 + 배경
- [ ] Warning (부분 추출 실패): 색상 + 배경
- [ ] Surface/Background: 팝업 배경, 섹션 배경
- [ ] 하드코딩 금지 목록: 모든 색상은 CSS 변수 or Tailwind 토큰으로만

### 3. 타이포그래피
- [ ] 상품명 표시: font-size, font-weight, max-lines (truncate 규칙)
- [ ] 가격: 강조 스타일 (금액은 굵게, 원화 단위 포함)
- [ ] 필드 라벨: 크기, 색상, 대/소문자 규칙
- [ ] 에러/힌트 메시지: 크기 + 색상

### 4. 레이아웃 구조 (ASCII 필수)
- [ ] 팝업 영역 구분: 헤더 / 메인 콘텐츠 / 액션 버튼 영역
- [ ] 각 영역의 height 소유자: 고정 vs 가변
- [ ] 내부 padding 소유자: 각 섹션이 스스로 갖는지, 래퍼가 갖는지

### 5. 상태별 UI 레이아웃 (익스텐션 전용)
- [ ] IDLE 상태: 팝업을 열었지만 아직 수집 전
- [ ] LOADING 상태: DOM 파싱 중 (스피너 위치, 메시지 포맷)
- [ ] SUCCESS 상태: 데이터 추출 완료 (필드 목록 표시)
- [ ] ERROR 상태: 파싱 실패 or 전송 실패
- [ ] SENDING 상태: API 전송 중
- [ ] SENT 상태: 전송 완료 확인

### 6. 컴포넌트 가이드
- [ ] FieldRow: 필드명 + 추출값 표시 컴포넌트 — props, truncate 규칙
- [ ] StatusBadge: IDLE/LOADING/SUCCESS/ERROR 상태 뱃지
- [ ] ActionButton: 수집 / 전송 / 재시도 — variant 규칙
- [ ] SectionCard: 상품정보/옵션/리뷰 각 섹션 래퍼
- [ ] EndpointChip: 연결된 엔드포인트 표시 소형 컴포넌트

### 7. UX 라이팅 (비개발자 타겟 기준)
- [ ] 버튼 라벨: "데이터 수집", "전송", "재시도", "설정" — 동의어 혼용 금지
- [ ] 상태 메시지: "수집 중...", "✓ 전송 완료", "전송 실패 — 다시 시도하세요"
- [ ] 에러 포맷: 원인 + 해결 행동 (e.g. "API 키가 유효하지 않습니다. 설정에서 확인해 주세요.")
- [ ] 빈 상태: 상품 페이지가 아닌 경우 표시할 메시지

### 8. 금지 패턴 카탈로그
- [ ] 팝업 안에서 modal 사용 금지 (팝업 위에 팝업은 UX 혼란)
- [ ] 하드코딩 색상 금지 (#fff, rgb() 직접 사용)
- [ ] 고정 height 없는 스크롤 영역 금지 (팝업이 무한정 늘어남)
- [ ] 팝업 밖 DOM 조작 없이 content script 결과만 수신
```

## 2. 브라우저 시각 검증 프로토콜 (익스텐션 팝업 특화)

### 검증 환경 셋업

```
필수 조건:
1. Chrome에 익스텐션 로드 (chrome://extensions → 개발자 모드 → 압축 해제)
2. 아래 테스트 쇼핑몰 탭 3개 이상 열기
   - 쿠팡 상품 페이지 (이미지 多, 옵션 복잡)
   - 네이버쇼핑 상품 페이지
   - 일반 쇼핑몰 상품 페이지 (Cafe24 or 메이크샵 기반)
3. 각 탭에서 팝업을 열어 검증
```

### 검증 타이밍

| 시점 | 검증 방법 | 필수 여부 |
|------|----------|----------|
| Sprint 1 UI 구현 후 | 팝업 직접 열어 레이아웃 확인 | 필수 |
| 각 상태(IDLE/LOADING/SUCCESS/ERROR) 구현 후 | 상태 전환 시 시각 확인 | 필수 |
| 데이터 추출 기능 연결 후 | 실제 쇼핑몰 3곳 이상 | 필수 |
| 스프린트 완료 시 | 전체 화면 + 엣지케이스 순회 | 필수 |
| Evaluator 평가 시 | 레퍼런스 대비 비교 스크린샷 | 필수 |

### 팝업 화면별 검증 체크리스트

```markdown
## 시각 검증 — 팝업 메인 화면

### 캔버스
- [ ] 팝업 너비가 설계 스펙(360px or 400px)인가?
- [ ] 팝업 높이가 콘텐츠에 따라 적절히 조정되는가?
- [ ] 팝업 전체가 브라우저 뷰포트 안에 들어오는가? (화면 밖으로 밀리지 않음)

### IDLE 상태 (수집 전)
- [ ] 수집 버튼이 주요 액션으로 명확히 강조되어 있는가?
- [ ] 연결된 엔드포인트 정보가 보이는가?
- [ ] 상품 페이지가 아닌 경우 안내 메시지가 표시되는가?

### LOADING 상태 (DOM 파싱 중)
- [ ] 로딩 스피너 위치가 IDLE 버튼 위치와 점프 없이 전환되는가?
- [ ] 스피너 + 메시지 조합이 팝업 중앙에 정렬되는가?
- [ ] 사용자가 이 상태에서 버튼을 다시 누를 수 없는가? (disabled)

### SUCCESS 상태 (데이터 추출 완료)
- [ ] 상품명이 truncate 규칙대로 잘리는가? (1~2줄 제한)
- [ ] 가격이 금액 포맷(원화 단위 포함)으로 표시되는가?
- [ ] 이미지 URL이 있으면 썸네일이 표시되는가?
- [ ] 옵션/스펙이 여러 개일 때 스크롤이 작동하는가?
- [ ] 리뷰 섹션이 접기/펼치기로 공간을 절약하는가?
- [ ] "전송" 버튼이 명확히 보이는가? (스크롤 없이도 접근 가능한가?)

### ERROR 상태 (파싱 실패 or 전송 실패)
- [ ] 에러 타입에 따라 메시지가 다르게 표시되는가?
- [ ] 에러 원인 + 해결 행동이 비개발자 언어로 표시되는가?
- [ ] 재시도 버튼이 명확히 제공되는가?

### SENDING / SENT 상태
- [ ] 전송 중 버튼이 비활성화되는가?
- [ ] 전송 완료 후 성공 상태가 명확히 표시되는가?
- [ ] 성공 후 다음 행동(팝업 닫기, 다른 상품 이동) 안내가 있는가?
```

```markdown
## 시각 검증 — 설정 화면 (엔드포인트/인증 토큰/필드 설정)

### 레이아웃
- [ ] 설정 화면으로 전환 시 팝업 크기 점프가 없는가?
- [ ] 입력 필드가 팝업 너비 안에서 온전히 표시되는가?
- [ ] 저장 버튼이 스크롤 없이 보이는가?

### 폼 인터랙션
- [ ] 필드 라벨 + 입력창 + 힌트/에러 순서가 일관되는가?
- [ ] 유효성 에러가 해당 필드 바로 아래에 표시되는가?
- [ ] API 키 입력 필드가 마스킹(password type)인가?
- [ ] 저장 후 피드백(토스트 or 인라인 확인)이 표시되는가?
```

## 3. 반복되는 시각 실패 패턴 (익스텐션 특화)

### 패턴 1: 팝업 높이 제한 없는 스크롤 영역
추출 데이터가 많을 때 팝업이 브라우저 밖으로 밀림.
**규칙**: 메인 콘텐츠 영역은 `max-height: calc(100vh - [헤더+버튼 고정높이]px); overflow-y: auto;`

### 패턴 2: 상태 전환 시 레이아웃 점프
IDLE → LOADING → SUCCESS 전환 시 높이가 급격히 바뀜.
**규칙**: 각 상태의 컨테이너는 `min-height`를 공유하거나 CSS transition 적용.

### 패턴 3: 상품명/가격 오버플로우
긴 상품명이 팝업을 가로로 밀어냄.
**규칙**: 상품명은 반드시 `overflow-hidden text-ellipsis` + `max-width: 100%` 적용.

### 패턴 4: 필드 선택 토글 미러링 불일치
설정에서 특정 필드를 끄면 결과 화면에서도 해당 필드가 사라져야 하는데 여전히 표시됨.
**규칙**: 수집 결과 렌더링 전 항상 `selectedFields` 필터링 로직 검증.

### 패턴 5: 실제 쇼핑몰에서만 재현되는 DOM 파싱 오류
단순 HTML 목업에서는 정상 동작, 실제 쇼핑몰에서는 SPA 구조로 인해 타이밍 이슈 발생.
**규칙**: QA는 반드시 실제 쇼핑몰 탭에서 수행. 목업 테스트만으로 완료 처리 금지.

### 패턴 6: 병렬 에이전트 디자인 드리프트
여러 에이전트가 팝업 UI / content script / 설정 화면을 동시 작업 → 컴포넌트 스타일 불일치.
**규칙**: 팝업 UI 스타일 작업은 순차 실행. 레퍼런스 화면(IDLE 상태) 먼저 완성 후 나머지 상태 복제.

## 4. 레퍼런스 기반 개발 (팝업 전용)

```
1. 팝업 IDLE 상태를 레퍼런스로 먼저 완성한다
2. 실제 Chrome에서 3개 쇼핑몰 탭에 띄워 검토/승인받는다
3. LOADING → SUCCESS → ERROR → SENT 순으로 레퍼런스 복제 방식으로 구현한다
4. 설정 화면은 별도 레퍼런스로 승인받고 구현한다
5. 각 상태 완성 후 실제 쇼핑몰에서 전환 흐름을 브라우저로 확인한다
```

## 5. 자동 검증 도구

| 도구 | 용도 | 비고 |
|------|------|------|
| `grep` 검증 | 하드코딩 색상(`#`, `rgb(`) 탐지 | 빌드 전 실행 |
| Vitest | 데이터 추출 로직 유닛 테스트 | 시각 QA 대체 불가 |
| Playwright | 팝업 UI E2E 테스트 | Chrome extension 전용 설정 필요 |
| 수동 브라우저 확인 | 실제 쇼핑몰 탭에서 팝업 직접 확인 | **시각 QA의 유일한 충분 조건** |

> Playwright의 Chrome Extension 지원은 제한적이다. `--disable-extensions` 없이 실행해야 하며, `chrome.storage` mock이 필요할 수 있다. 자동 테스트에 과의존하지 말고 수동 확인을 병행한다.

## 6. 멀티 페르소나 QA 적용 시

| 이 문서의 항목 | 담당 페르소나 | 역할 |
|--------------|------------|------|
| 디자인 가이드 준수 | 디자이너 | 토큰/레이아웃/타이포 일관성 |
| 팝업 상태별 검증 | 디자이너 + 사용자 | 디자이너: 일관성 / 사용자: 비개발자 관점 혼란 여부 |
| 실제 쇼핑몰 데이터 추출 흐름 | 사용자 (비개발자 MD 페르소나) | "클릭 한 번"이 실제로 그만큼 쉬운가? |
| 오류 메시지 이해도 | 사용자 페르소나 | 에러 메시지를 읽고 스스로 해결할 수 있는가? |
| DOM 파싱 정확도 | 개발자 + 사용자 | 개발자: 추출 정확도 / 사용자: 잘못 추출된 값 발견 |

**사용자 페르소나 필수 시나리오**: 쿠팡 → 상품 페이지 이동 → 팝업 클릭 → 수집 → 전송까지 전체 플로우를 막힘 없이 완수할 수 있는가.
