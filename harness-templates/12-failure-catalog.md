# 실패 카탈로그 — 쇼핑몰 데이터 수집 크롬 익스텐션

> Chrome Extension + 범용 DOM 파서 프로젝트에서 반복될 가능성이 높은 실패 패턴.
> Sprint 0 착수 전 팀 전체가 읽고 체크할 것.

---

## 실패 유형 1: 자기 평가 편향 (Self-Evaluation Bias)

**이 프로젝트에서의 증상**: DOM 파서를 작성한 에이전트가 "쿠팡/네이버쇼핑에서 테스트 완료"라고 보고했으나, 실제 사용 시 가격 필드 미추출, 옵션 드롭다운 미인식

**근본 원인**: 코드를 작성한 에이전트가 자기 코드를 평가 → "셀렉터가 있으니 되겠지" 합리화

**해결**: 파서 작성자 ≠ 파서 검증자. 반드시 **별도 에이전트**가 실제 쇼핑몰 10곳에서 교차 검증.

### 자기 진단
- DOM 파서를 만든 에이전트가 "완료"를 선언하는가? → 독립 검증 에이전트 분리 필요
- "빌드 성공 = 데이터 정상 추출"로 판단하는가? → 실제 상품 페이지 열어 팝업에서 확인
- 1개 쇼핑몰에서만 테스트했는가? → 최소 5개 구조가 다른 쇼핑몰에서 Evidence 확보

---

## 실패 유형 2: 양적 착시 (Quantity Illusion)

**이 프로젝트에서의 증상**: 12개 쇼핑몰 파서를 Sprint에 넣었으나, 실제로 완전히 동작하는 것은 2~3개

**근본 원인**: "지원 쇼핑몰 수"에 집중하고 "각 쇼핑몰에서 모든 필드가 실제로 추출되는가"를 확인하지 않음

**해결**: 스프린트당 신규 쇼핑몰 지원은 **3개 이하**. **3개 완전체 > 12개 반제품**.

### 자기 진단
- Sprint에 신규 쇼핑몰 지원이 5개 이상인가? → 범위 축소
- "일단 셀렉터 넣고 나중에 검증"인가? → 추가 즉시 해당 쇼핑몰 전 필드 확인
- 완료 기준이 "파서 코드 존재"인가? → "팝업에서 미리보기 정상 표시"로 변경

---

## 실패 유형 3: 코드 QA ≠ 실사용 QA

**이 프로젝트에서의 증상**: 빌드 통과했지만 팝업 미리보기에서 이미지 URL 깨짐, 가격에 통화 기호 포함된 문자열, 리뷰 개수 0 표시

**근본 원인**: 유닛 테스트는 mock DOM에서 통과하지만, 실제 쇼핑몰 페이지의 lazy-load·동적 렌더링·Shadow DOM은 모름

**해결**: 실제 Chrome에서 익스텐션 로드 후 **팝업 열어서 미리보기 육안 확인** 필수. 코드 QA + 실사용 QA 양쪽 통과해야 PASS.

### 자기 진단
- QA가 Vitest 유닛 테스트만으로 이루어지는가? → 실제 Chrome + 상품 페이지 확인 추가
- Mock DOM과 실제 DOM이 다를 수 있는 경우를 고려했는가? → lazy-load 대기 로직 확인
- 디자인 가이드 없이 팝업 UI를 만드는가? → `design-guide.md` 선작성 후 구현

---

## 실패 유형 4: 병렬 에이전트 디자인 드리프트

**이 프로젝트에서의 증상**: 팝업 메인 화면과 설정 화면을 동시에 개발 → 같은 컴포넌트인데 버튼 스타일, 폰트 크기, 색상 코드 불일치

**근본 원인**: 에이전트 간 중간 결과물 공유 불가. 팝업 240px 너비 제약을 각자 다르게 해석.

**해결**: 팝업 UI 통일 작업은 **순차 실행**. 팝업 메인 화면 레퍼런스 1개 완성 → 검토 → 설정·상태 화면 복제.

### 자기 진단
- 팝업 메인·설정·상태 화면을 동시에 에이전트에 맡기는가? → 메인 먼저, 나머지는 복제
- 레퍼런스 컴포넌트 없이 여러 화면을 병렬로 만드는가? → 레퍼런스 먼저
- Chrome Extension 240px 너비 제약이 명세에 명시되어 있는가? → `design-guide.md`에 고정 기입

## 실패 유형 5: 감사 순서 위반

**이 프로젝트에서의 증상**: 전송 포맷 설정 기능을 개발 → 이후 인증 토큰 관리 코드 감사 → 토큰 저장 방식 충돌 발견 → 재작업

**근본 원인**: 기획 → 개발 → 감사 순서 (올바른 순서: **감사 → 범위 확정 → 기획**)

**해결**: Sprint마다 **기존 Content Script·Background Worker·Popup 코드 감사 먼저**, 그 다음 신규 기능 범위 확정.

### 자기 진단
- 현재 manifest.json·background.ts·content.ts 상태를 파악하지 않고 새 기능을 계획하는가? → 감사 먼저
- "일단 만들고 나중에 통합하자"인가? → Chrome Storage 스키마 충돌 먼저 확인

## 실패 유형 6: 공통 로직 중복 실행

**이 프로젝트에서의 증상**: Content Script와 Popup이 각각 독립적으로 DOM 파싱 실행 → 동일 페이지에서 파싱이 2번 돌아 느린 응답

**근본 원인**: Content Script ↔ Popup 간 메시지 패싱 설계 미흡. 각자 파싱을 독립 실행.

**해결**: 파싱은 **Content Script 단독 실행**, 결과를 Popup이 `chrome.runtime.sendMessage`로 수신하는 단방향 구조 확정 후 착수.

### 자기 진단
- Content Script와 Popup 양쪽에 파싱 로직이 있는가? → Content Script로 단일화
- 팝업 열 때마다 전체 DOM을 다시 파싱하는가? → 세션 캐시 또는 Background 캐싱 적용
- 메시지 패싱 인터페이스가 타입으로 정의되어 있는가? → TypeScript interface로 먼저 정의

---

## 실패 유형 7: 이중 래핑 / 권한 오염

**이 프로젝트에서의 증상**: `content_scripts` 와 `web_accessible_resources` 양쪽에 동일 파일 등록 → 충돌. 팝업에서 직접 `fetch` 실행 → CSP 차단.

**근본 원인**: 다른 시점에 다른 에이전트가 manifest.json을 각각 수정 → 서로의 권한 설정을 모름

**해결**: manifest.json 권한·파일 등록 소유자를 `CLAUDE.md`에 명시. **manifest.json은 한 에이전트만 수정**, 나머지는 PR로 요청.

### 자기 진단
- manifest.json을 여러 에이전트가 동시에 수정하는가? → 단일 소유자 지정
- Popup에서 직접 외부 API를 fetch하는가? → Background Worker를 통해 우회
- `host_permissions`에 `<all_urls>` 추가 이유가 문서화되어 있는가? → 변경 사유 주석 필수

## 이 프로젝트 전용 추가 실패 유형

### 실패 유형 8: 범용 파서의 범용성 착각

**증상**: "범용 DOM 파서"가 실제로는 특정 쇼핑몰 구조에만 최적화되어 있음

**근본 원인**: 개발 중 가장 많이 열었던 쇼핑몰(예: 쿠팡)에 과적합. 다른 구조(오픈마켓, 브랜드몰, 해외몰)에서 추출 실패.

**해결**: 파서 설계 시 **구조 유형 분류표** 먼저 작성 (그리드형/상세형/탭형 등). 유형별 최소 2개 쇼핑몰에서 검증.

### 실패 유형 9: 인증 토큰 평문 저장

**증상**: `chrome.storage.local`에 API 토큰 평문 저장 → 보안 감사에서 지적

**근본 원인**: 빠른 개발을 위해 암호화 생략. "나중에 하자"가 출시 직전까지 미뤄짐.

**해결**: Sprint 0에서 토큰 저장 방식 확정. `chrome.storage.session` 우선 검토, 평문 저장 시 명시적 승인 필요.

## 생산:수리 비율의 교훈

크롬 익스텐션 프로젝트에서 관찰되는 특수 비율:
- **파서 초기 구현**: 1x 시간
- **쇼핑몰별 예외 처리·수리**: 2~3x 시간 (예상보다 항상 많음)
- **manifest 권한 충돌 디버깅**: 예상 외 병목

**교훈**: "일단 범용으로 만들고 나중에 예외 추가"는 이 프로젝트의 가장 비싼 기술 부채. Sprint 0에서 파서 아키텍처와 manifest 소유 규칙을 확정하는 것이 전체 일정을 결정한다.
