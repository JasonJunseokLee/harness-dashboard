# 07. 비용/시간 트레이드오프 — 쇼핑몰 데이터 수집 크롬 익스텐션 맞춤 가이드

> 이 프로젝트는 "작동하지 않으면 즉시 신뢰를 잃는" B2B 도구입니다.
> 비개발자 운영자가 실무에서 직접 사용하므로, **품질 실패 비용이 비용 절감보다 훨씬 큽니다.**

---

## 이 프로젝트의 실제 비용 구조 추정

### 스프린트별 권장 하네스 수준

| Sprint | 내용 | 권장 수준 | 예상 비용 | 이유 |
|--------|------|-----------|-----------|------|
| Sprint 0 | 프로젝트 셋업, 기술 스택 확정 | Level 0 | $1~5 | 표준 보일러플레이트, 창의성보다 정확성 |
| Sprint 1 | 범용 DOM 파서 MVP | **Level 2** | $100~200 | 핵심 알고리즘, 쇼핑몰 다양성 대응이 품질 좌우 |
| Sprint 2 | 팝업 UI + 추출 미리보기 | Level 1 | $50~100 | UI는 반복 수정 가능, QA 2라운드면 충분 |
| Sprint 3 | API 전송 + 인증 토큰 관리 | **Level 2** | $80~150 | 보안·데이터 무결성은 스펙 오류 허용 불가 |
| Sprint 4 | 옵션/스펙/리뷰 파서 확장 | Level 1 | $60~120 | DOM 패턴 추가, 구조적이지만 반복 허용 |
| Sprint 5 | 자동 전송, 필드 선택, 포맷 설정 | Level 1 | $50~80 | 기능 추가 레이어, 코어 안정화 이후 |
| Sprint 6 | 안정화, 크로스-쇼핑몰 테스트 | Level 1 QA 집중 | $40~80 | 새 구현 없음, QA 라운드만 집중 |

## 이 프로젝트에 특히 중요한 판단 원칙

### DOM 파서는 반드시 Level 2

범용 DOM 파싱은 이 익스텐션의 **핵심 가치 제안**입니다.

```
스펙 오류 시 발생하는 문제:
├── 쿠팡에서 작동 → 네이버쇼핑에서 필드 누락
├── 가격 파싱 오류 → 고객사 데이터 오염
└── 리뷰 수집 실패 → "이 툴 못 쓰겠다" 즉각 이탈
```

→ Planner가 쇼핑몰 유형별 DOM 패턴을 사전 스펙화하고,
   Evaluator가 최소 3개 쇼핑몰(쿠팡·네이버·무신사 등)로 크로스 검증해야 합니다.

### 인증·전송 레이어도 Level 2

B2B 운영자의 API 키와 엔드포인트를 다루는 코드입니다.

```
스펙 오류 시 발생하는 문제:
├── 토큰 노출 → 보안 사고
├── 중복 전송 → 고객사 데이터 오염
└── 전송 실패 무통보 → 운영자가 인지 불가
```

---

## 숨겨진 비용: 이 프로젝트에서 더 중요한 이유

Solo Agent로 DOM 파서를 짰을 때 실제 TCO:

| 비용 항목 | Solo Agent | Level 2 Harness |
|-----------|-----------|-----------------|
| API 비용 | $10~15 | $100~200 |
| 쇼핑몰별 파서 수정 (수동) | 쇼핑몰당 $20~50 × N곳 | 거의 없음 |
| 고객사 데이터 오류 대응 | 건당 수시간 | 거의 없음 |
| **실질 TCO** | **높음** | **예측 가능** |

> "쇼핑몰 구조가 바뀔 때마다 크롤러가 깨진다"는 기존 문제를 **이 익스텐션이 그대로 재현하면 안 됩니다.**

## 비용 최적화 팁 — 이 프로젝트 한정

1. **Planner에 쇼핑몰 커버리지 목록을 명시적으로 주입하세요**
   - "쿠팡·네이버·11번가·무신사·올리브영" 등 목표 사이트를 선언하면 Planner 비용 $0.5 추가로 Build 재작업 3~4시간을 절약합니다.

2. **QA 라운드는 실제 쇼핑몰 URL로 진행하세요**
   - 목 데이터가 아닌 실제 상품 페이지에서 Evaluator가 검증해야 의미 있습니다.
   - Mock DOM 기반 QA는 "테스트는 통과하지만 실제는 실패"하는 Chrome Extension의 고전적 함정입니다.

3. **UI Sprint는 Level 1으로 충분합니다**
   - 팝업 UI(미리보기, 상태 표시)는 반복 수정 비용이 낮고 사용자 피드백으로 조정 가능합니다.
   - 여기서 아낀 비용을 DOM 파서와 전송 레이어에 집중 투자하세요.

4. **Sprint 1 이전에 기술 스택을 반드시 확정하세요**
   - "미정"인 상태로 Level 2 하네스를 돌리면 Planner가 잘못된 전제로 스펙을 생성합니다.
   - 권고 스택(MV3 + TypeScript + React + Vite)을 Sprint 0에서 확정 → Sprint 1 Level 2 시작.

## 의사결정 체크리스트 — 이 프로젝트용

Sprint 시작 전 확인:

- [ ] 이 Sprint의 산출물이 "고객사 데이터에 직접 영향"을 주는가? → Yes면 Level 2
- [ ] DOM 파싱 로직이 포함되어 있는가? → Yes면 Level 2, Evaluator에 실제 쇼핑몰 URL 제공
- [ ] 보안(토큰·API키) 관련 코드가 포함되어 있는가? → Yes면 Level 2
- [ ] UI/설정 레이어만 다루는가? → Level 1로 충분
- [ ] 기술 스택이 확정되어 있는가? → 미확정이면 Level 2 시작 불가
- [ ] Evaluator가 테스트할 실제 쇼핑몰 목록을 준비했는가?
- [ ] QA 라운드는 2~3회로 제한했는가? (그 이상은 수확 체감)
