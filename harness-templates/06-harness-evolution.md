# 06. 하네스 진화 전략 — 쇼핑몰 데이터 수집 크롬 익스텐션 맞춤 버전

> "모든 하네스 컴포넌트는 현재 모델의 한계를 가정한다. DOM 파싱 로직이 안정되고 모델이 개선될수록, 하네스도 함께 가벼워져야 한다."

---

## 핵심 원칙

이 프로젝트의 하네스는 두 가지 불확실성을 보완하기 위해 존재한다:
1. **기술 스택 미확정** — Sprint 0에서 확정되기 전까지 설계 결정이 많다
2. **범용 DOM 파싱의 품질 편차** — 쇼핑몰마다 HTML 구조가 달라 에이전트 출력 품질이 불안정할 수 있다

하네스가 과도하면 속도를 잡아먹고, 부족하면 DOM 파싱 오류를 놓친다. **스프린트별로 load-bearing 여부를 재검토**하라.

---

## 이 프로젝트의 하네스 구성 현황

| 컴포넌트 | 추가 이유 | 보완하는 한계 | 현재 상태 |
|----------|----------|------------|---------|
| Planner 에이전트 | 기술 스택 미확정, 범용 파서 설계 복잡 | 아키텍처 오결정 방지 | **필수** (Sprint 0~1) |
| Evaluator 에이전트 | DOM 파싱 결과 품질 편차 | 쇼핑몰별 추출 실패 감지 | **필수** (Sprint 1~4) |
| 스프린트 분해 (7개) | 쇼핑몰 커버리지 확대가 점진적으로 이루어짐 | 한 번에 전체 구현 시 테스트 불가 | **필수** |
| 전송 포맷 계약 (스프린트 계약) | 엔드포인트 연동 스펙이 팀 간 오해를 유발할 수 있음 | 범위 이탈 방지 | **필수** |
| Few-shot 캘리브레이션 | 쇼핑몰별 DOM 예시가 평가 일관성에 영향 | 평가 기준 흔들림 | 테스트 필요 |

---

## 컴포넌트 부하 테스트 — 이 프로젝트 적용 기준

### 테스트 방법

```
1. 컴포넌트 X를 제거하고 동일 스프린트 작업 수행
2. 결과물 품질 비교
   ├── 품질 저하 없음 → X는 불필요, 다음 스프린트부터 제거
   ├── 약간 저하 → 복잡도 대비 판단 (예: DOM 파싱 정확도 5% 미만 하락이면 제거)
   └── 크게 저하 → load-bearing, 유지
3. 다음 모델 업그레이드 또는 스프린트 전환 시 재테스트
```

### 이 프로젝트 전용 테스트 대상

- [ ] **Planner 에이전트** — 기술 스택이 Sprint 0에서 확정된 이후에는 필요성 재검토. 확정 후 Sprint 2부터 제거 가능한지 테스트
- [ ] **스프린트 분해** — Sprint 3(주요 쇼핑몰 커버리지) 이후 모델이 멀티-사이트 파싱을 한 번에 안정적으로 구현할 수 있다면 스프린트 통합 고려
- [ ] **Evaluator 에이전트** — 네이버쇼핑·쿠팡·11번가 등 주요 3개 사이트에서 파싱 성공률 95% 이상이면 단일 패스 검증으로 전환 검토
- [ ] **Few-shot 캘리브레이션** — 쇼핑몰 DOM 예시 없이도 평가 기준이 일관되는지 Sprint 2 완료 후 확인
- [ ] **전송 포맷 계약** — 엔드포인트 스펙이 고정된 Sprint 3 이후 필요 여부 재검토

---

## 진화 프레임워크 — 스프린트별 적용

### Sprint 0~1: 기반 구축기
- Planner + Evaluator **모두 필수**
- 기술 스택(MV3, React/Vite, Rollup 등) 결정이 이후 모든 컴포넌트에 영향
- 이 시기에 하네스 컴포넌트를 줄이지 마라 — 아키텍처 오결정 비용이 너무 크다

### Sprint 2~4: 파싱 안정화기
- Evaluator는 **쇼핑몰 커버리지 테스트**에 집중
- 파싱 성공률 지표를 트래킹해서 Evaluator 제거 가능 시점을 객관적으로 판단
- Planner는 제거 후보 — 스프린트 계획이 반복 패턴으로 자리잡으면 단순화

### Sprint 5~6: 안정화·배포기
- 하네스 최소화 목표
- 반복 검증이 완료된 컴포넌트부터 순서대로 제거
- 최종적으로 Evaluator만 남기는 것이 이상적인 엔드스테이트

---

## 이 프로젝트에서의 중요한 통찰

### DOM 파싱 품질 편차가 핵심 변수다

> "범용 DOM 파서의 성공률이 높아질수록 Evaluator의 역할은 '오류 감지'에서 '데이터 품질 보증'으로 이동한다."

- 초기: Evaluator가 파싱 실패 자체를 잡아야 함 → 촘촘한 평가 기준 필요
- 성숙기: 파싱은 안정적, Evaluator는 데이터 정합성(가격 포맷, 이미지 URL 유효성 등)만 확인 → 경량화 가능

### 쇼핑몰 커버리지 확대는 하네스 재설계 트리거다

스프린트마다 지원 쇼핑몰이 늘어나면 이전 평가 기준이 맞지 않을 수 있다.
**새 쇼핑몰 유형 추가 시마다 Evaluator의 few-shot 예시와 평가 기준을 업데이트**하라.

## 모델 업그레이드 시 체크리스트 (이 프로젝트 전용)

새 Claude 버전 또는 새 스프린트 전환 시:

- [ ] 현재 DOM 파싱 성공률 측정 (네이버쇼핑, 쿠팡, 11번가 기준)
- [ ] Planner 없이 스프린트 목표를 Generator에게 직접 지시해도 품질이 유지되는가?
- [ ] Evaluator 없이 Generator 자체 점검만으로 파싱 오류를 잡을 수 있는가?
- [ ] 스프린트 분해 없이 두 스프린트를 합쳐도 결과물 품질이 유지되는가?
- [ ] 인증 토큰 관리·전송 포맷·엔드포인트 등록 기능이 안정화된 이후 전송 계약이 여전히 필요한가?
- [ ] 위 결과를 문서화하여 다음 검토 시 비교 기준으로 보존

---

> **요약**: 이 프로젝트는 Sprint 0~1에서 하네스를 최대로 유지하고, Sprint 3 이후 파싱 성공률 지표를 기준으로 컴포넌트를 순차적으로 제거한다. 복잡성은 "아키텍처 결정 불확실성 → 쇼핑몰 커버리지 편차 → 데이터 품질 보증" 순서로 이동하며, 하네스도 그 흐름에 맞춰 재설계한다.
