# 01. 핵심 원칙 5가지 — 쇼핑몰 데이터 수집 크롬 익스텐션 맞춤 버전

> Chrome MV3 익스텐션 × 범용 DOM 파서 × 비개발자 타겟 프로젝트에 특화된 하네스 설계 원칙

---

## 원칙 1: 작업을 다루기 쉬운 단위로 분해하라

**이 프로젝트에서 왜 특히 중요한가?**
익스텐션은 Content Script / Background Service Worker / Popup UI / Options Page가 서로 메시지로 통신하는 **멀티 컨텍스트 구조**다. 레이어를 한꺼번에 개발하면 메시지 버스 오류가 어디서 왔는지 추적이 불가능해진다.

**적용 방법:**
- 스프린트 0~6을 **레이어 경계 기준**으로 분리 유지 (계획대로)
- 각 스프린트는 **하나의 Chrome API 도메인**에만 집중
  - Sprint 1: DOM 파싱 (Content Script)
  - Sprint 2: 데이터 전송 (Background + fetch)
  - Sprint 3: Popup UI
  - Sprint 4: Storage / 설정
  - Sprint 5: 쇼핑몰 커버리지 확대
  - Sprint 6: 안정화 / E2E
- 스프린트 시작 전 **Contract 정의**: "이번 스프린트에서 건드리는 Chrome API는 `X`만이다"

**예시:**
```
❌ "팝업 UI 만들면서 DOM 파서도 연결하고 API 전송도 구현해줘"
✅ Sprint 1 계약: "Content Script에서 상품명·가격·이미지만 추출, 콘솔 로그 출력까지. 전송은 다음 스프린트"
```

---

## 원칙 2: 구조화된 아티팩트로 컨텍스트를 전달하라

**이 프로젝트에서 왜 특히 중요한가?**
Chrome 익스텐션은 Content Script ↔ Background ↔ Popup 간 통신이 **비동기 메시지 패싱**으로만 이루어진다. 에이전트 세션 간 핸드오프도 마찬가지—구조 없이 구두로 전달하면 메시지 포맷이 세션마다 달라진다.

**적용 방법:**
- `.harness/` 폴더 아래 스프린트별 핸드오프 문서 유지
- Chrome 메시지 포맷을 **계약 파일**로 명시 (e.g. `message-contracts.ts`)
  ```ts
  // message-contracts.ts — 세션 간 절대 변경 전 합의 필요
  type ExtractRequest = { type: 'EXTRACT'; tabId: number }
  type ExtractResponse = { ok: boolean; data?: ProductData; error?: string }
  ```
- DOM 파서가 추출한 `ProductData` 스키마도 파일로 고정 후 레이어 간 공유
- 세션 리셋 시 핸드오프 문서에 반드시 포함:
  1. 현재 동작하는 쇼핑몰 목록 및 파싱 성공률
  2. 깨진 선택자(Selector)와 원인
  3. 다음 스프린트에서 건드릴 Chrome API
  4. 현재 메시지 포맷 버전

---

## 원칙 3: 생성과 평가의 관심사를 분리하라

**이 프로젝트에서 왜 특히 중요한가?**
범용 DOM 파서는 **쇼핑몰 A에서 잘 되면 쇼핑몰 B에서 망가지는** 패턴이 반복된다. Generator가 "됩니다"라고 말해도 실제 사이트에서 Evaluator가 직접 검증해야 신뢰할 수 있다.

**적용 방법:**
- **Generator**: DOM 선택자 로직·파싱 알고리즘·API 전송 코드 작성에만 집중
- **Evaluator**: Playwright로 실제 쇼핑몰 페이지를 열어 추출 결과 검증
  ```
  검증 대상 (최소 3개 쇼핑몰 커버):
  - 쿠팡 상품 페이지
  - 네이버 스마트스토어
  - 11번가
  ```
- Evaluator는 **추출 미리보기** 화면 기준으로 채점 (사용자가 보는 것과 동일 관점)
- 같은 세션에서 파서 코드를 짜고 "잘 됩니다"라고 자체 평가하는 것 금지

## 원칙 4: 주관적 기준을 채점 가능하게 변환하라

**이 프로젝트에서 왜 특히 중요한가?**
비개발자 타겟이므로 "쓰기 쉬운가"가 핵심 품질 기준이다. 그러나 이 기준은 흔히 주관적으로 흘러간다. 채점 기준 없이 평가하면 매 스프린트마다 판단이 달라진다.

**채점 기준표:**

| 영역 | 평가 질문 | 임계값 |
|------|----------|--------|
| 팝업 UX | 클릭 3회 이내로 수집→전송 완료 가능한가? | 8/10 |
| DOM 파싱 정확도 | 상품명·가격 추출 성공률 (테스트 쇼핑몰 기준) | 90% |
| 오류 처리 | 추출 실패 시 사용자가 이해할 수 있는 메시지를 보여주는가? | 7/10 |
| 설정 명확성 | 비개발자가 가이드 없이 엔드포인트·토큰을 등록할 수 있는가? | 7/10 |
| 전송 신뢰성 | 전송 성공/실패가 팝업에서 명확히 표시되는가? | 8/10 |
| 성능 | 팝업 열기 → 추출 완료까지 3초 이내인가? | 필수 |

**임계값 미달 시:** 구체적 피드백(어느 쇼핑몰, 어느 필드, 어느 단계)과 함께 해당 스프린트 재작업

## 원칙 5: 각 컴포넌트의 존재 이유를 지속적으로 검증하라

**이 프로젝트에서 왜 특히 중요한가?**
Chrome MV3는 MV2 대비 Background Script가 Service Worker로 대체되어 **상태를 메모리에 유지하지 못한다**. 초기에 "필수"라고 설계한 인메모리 캐시나 상태 관리 컴포넌트가 MV3 제약에서 오히려 버그 원인이 될 수 있다.

**적용 방법:**
- 스프린트마다 아래 질문을 반드시 확인:
  - "이 컴포넌트가 Chrome Storage로 대체 가능한가?"
  - "Background까지 갈 필요 없이 Content Script에서 직접 처리 가능한가?"
  - "이 추상화 레이어가 현재 지원해야 하는 쇼핑몰 수에 비해 과도하지 않은가?"
- **부하 테스트 기준**: 쇼핑몰 5개 → 20개로 늘릴 때 파서 레이어를 교체 없이 확장 가능한가?
- Sprint 5(쇼핑몰 커버리지 확대) 진입 전 반드시 아키텍처 재검토
- 불필요한 메시지 패싱 홉(hop)은 과감히 제거 — Content Script → Popup 직접 통신이 가능하면 Background 경유 불필요

## 프로젝트 맞춤 체크리스트

스프린트 시작 전 아래를 확인:

- [ ] 이번 스프린트가 건드리는 Chrome API 도메인이 하나로 한정되어 있는가?
- [ ] Chrome 메시지 포맷이 `message-contracts.ts`에 명시되어 있는가?
- [ ] DOM 파서 변경 시 테스트 쇼핑몰 3개 이상에서 Evaluator 검증을 거쳤는가?
- [ ] 비개발자 사용자 관점에서 팝업 UX를 채점 기준표로 평가했는가?
- [ ] Background Service Worker에 상태를 보관하는 컴포넌트가 있다면 Chrome Storage로 대체 가능한지 검토했는가?
- [ ] 핸드오프 문서에 현재 파싱 성공률과 깨진 선택자 목록이 업데이트되어 있는가?
