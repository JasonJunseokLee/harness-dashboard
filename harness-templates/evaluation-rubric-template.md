# 평가 기준표 (Evaluation Rubric)

> 프로젝트 시작 시 작성하여 모든 스프린트에 걸쳐 일관된 평가를 보장하는 문서.
> 스프린트 계약서의 "평가 기준" 섹션은 이 기준표에서 해당 스프린트에 적용할 항목을 선택한다.

---

## 프로젝트 정보

| 항목 | 값 |
|------|---|
| 프로젝트명 | |
| 프로젝트 타입 | UI / API / CLI / SDK / 데이터파이프라인 / 기타 |
| 품질 수준 | MVP / 프로덕션 / 완성품 |
| QA 레벨 | Level 1 (단일 Evaluator) / Level 2 (3-페르소나) |

---

## 채점 영역

### 1. 기능성 (Functionality)

| 점수 | 기준 |
|------|------|
| 9-10 | 모든 기능이 완벽 동작 + 엣지케이스까지 우아하게 처리 |
| 7-8 | 핵심 기능 모두 동작 + 주요 엣지케이스 처리 |
| 5-6 | 핵심 기능은 동작하나 일부 시나리오에서 오류 |
| 3-4 | 기본 동작은 하나 주요 기능에 버그 |
| 1-2 | 핵심 기능이 동작하지 않음 |

**임계값**: {N}/10
**이 프로젝트에서의 의미**: {구체적으로 어떤 기능이 동작해야 하는지}

---

### 2. 코드 품질 (Code Quality)

| 점수 | 기준 |
|------|------|
| 9-10 | 기존 패턴 완벽 준수 + 읽기 쉬움 + 테스트 충분 + 보안 문제 없음 |
| 7-8 | 패턴 준수 + 합리적 구조 + 기본 테스트 + 알려진 보안 이슈 없음 |
| 5-6 | 대체로 괜찮으나 일부 패턴 불일치 또는 테스트 부족 |
| 3-4 | 읽기 어려움 또는 보안 취약점 존재 |
| 1-2 | 유지보수 불가능한 코드 |

**임계값**: {N}/10
**이 프로젝트에서의 의미**: {어떤 패턴을 따라야 하는지, 테스트 기준}

---

### 3. 제품 깊이 (Product Depth)

| 점수 | 기준 |
|------|------|
| 9-10 | 모든 상태(로딩/빈/에러) 구현 + 세부 디테일 정교 + 실사용 가능 |
| 7-8 | 주요 상태 구현 + 기본 디테일 처리 |
| 5-6 | 정상 경로만 구현 + 에러/빈 상태 미흡 |
| 3-4 | 표면적 구현 + 상태 처리 없음 |
| 1-2 | 데모 수준 — 실사용 불가 |

**임계값**: {N}/10
**이 프로젝트에서의 의미**: {어떤 상태들이 필수인지}

### 4. 시각 디자인 (Visual Design) — UI 프로젝트만

| 7-8 | design-guide 대체로 준수 + 사소한 불일치만 |
| 5-6 | 토큰 시스템 부분 준수 + 일부 하드코딩 |
| 3-4 | 일관성 부족 + 디자인 가이드 미준수 |
| 1-2 | 시각적으로 깨짐 |

**임계값**: {N}/10
**이 프로젝트에서의 의미**: {design-guide.md 참조}

### 5. DX 설계 (Developer Experience) — API/CLI/SDK만

| 점수 | 기준 |
|------|------|

## 합리화 방지 규칙

이 프로젝트의 Evaluator에게 적용되는 규칙:

```
금지:
- "전체적으로 잘 되어 있습니다" → 각 항목별로 구체적 점수를 매겨라
- "사소한 문제입니다" → 사소함의 기준은 임계값이다. 점수로 표현하라
- "나중에 고치면 됩니다" → 지금 이 스프린트 기준으로 판정하라
- "동작은 합니다" → 동작 여부는 기능성의 일부일 뿐이다

필수:
- 모든 점수에 구체적 근거를 1줄 이상 작성하라
- FAIL 항목에 파일경로:라인 수준의 증거를 포함하라
- 임계값 미달 항목에 구체적 수정 방향을 제시하라
```
