# 02. 아키텍처 패턴 — 쇼핑몰 데이터 수집 크롬 익스텐션 맞춤 버전

> Generator-Evaluator 루프와 3-에이전트 시스템의 이 프로젝트 적용 방법

---

## 이 프로젝트에 권장하는 하네스 수준: **Level 2 (3-에이전트)**

**이유**: 범용 DOM 파서 개발 + Chrome MV3 아키텍처 + 외부 API 연동이 복합된 중~고 복잡도 프로젝트. 쇼핑몰마다 HTML 구조가 달라 파싱 정확도 검증이 반드시 필요하며, 비개발자 사용자를 위한 UX 품질 기준이 높음.

---

## Generator-Evaluator 루프

### 기본 흐름

```
┌─────────────┐     결과물     ┌─────────────────────────────┐
│  Generator  │ ─────────────→ │  Evaluator                  │
│  (구현자)    │               │  (검증자: 실제 쇼핑몰에서    │
│             │ ←───────────── │   익스텐션 동작 확인)        │
└─────────────┘   피드백+점수   └─────────────────────────────┘
       ↑                              │
       └──────── 반복 (5~10회) ────────┘
```

### 이 프로젝트에서 Evaluator가 해야 하는 것

Evaluator는 코드를 읽는 것이 아니라 **실제 사용자처럼 익스텐션을 써야 한다**.

```
✅ 올바른 평가 방법
- Playwright로 쿠팡/네이버쇼핑/무신사 상품 페이지에 직접 접속
- 익스텐션 아이콘 클릭 → 팝업 열림 확인
- 추출된 데이터를 눈으로 검증 (상품명, 가격, 이미지 URL 정확성)
- 전송 버튼 클릭 → 모의 API 서버 수신 여부 확인
- 비개발자(MD, 운영자)가 설정 화면을 혼자 완결할 수 있는지 확인

❌ 잘못된 평가 방법
- 소스 코드 리뷰만 수행
- 특정 쇼핑몰 1개에서만 테스트
- "동작하는 것 같아 보인다"는 주관적 판단
```

### 핵심 포인트
- **범용성이 핵심**: 최소 3개 이상의 서로 다른 쇼핑몰에서 동작 검증 필수
- 반복 횟수는 스프린트당 5~10회 권장
- 각 반복마다 "어떤 쇼핑몰에서 어떤 필드가 틀렸는지" 구체적 피드백 필요

## 3-에이전트 시스템 (이 프로젝트 적용)

```
┌──────────┐    스펙    ┌──────────────┐   결과물   ┌──────────────────────┐
│ Planner  │ ────────→ │  Generator   │ ────────→ │  Evaluator           │
│ (기획자)  │          │  (구현자)     │           │  (실제 쇼핑몰 검증)   │
└──────────┘          └──────────────┘           └──────────────────────┘
                            ↑                          │
                            └──── 피드백 + 재작업 ───────┘
```

### 각 에이전트의 역할 (프로젝트 맞춤)

#### Planner (기획자)
- **입력**: "쿠팡 상품 페이지에서 가격 추출 기능 추가"
- **출력**: DOM 파싱 전략, 데이터 스키마, 예외 케이스(품절/옵션 없음 등) 포함 스펙
- **집중**: 어떤 데이터를 어떤 구조로 추출할지 (구현 방법은 Generator에 위임)
- **금지**: `querySelector('div.price')` 같은 구현 세부사항을 직접 지시하지 않음
- **특히 주의**: 쇼핑몰마다 동일 데이터가 다른 DOM 경로에 있을 수 있음 → 스펙에 폴백 전략 포함

#### Generator (구현자)
- Chrome MV3 규격 준수 (Manifest V2 패턴 사용 금지)
- Popup UI(React + Vite)와 Content Script 간 메시지 통신 구조 유지
- 스프린트 단위 구현: 한 번에 하나의 쇼핑몰 혹은 하나의 기능
- 자체 점검: `chrome.runtime.sendMessage` 오류, CORS 이슈, CSP 위반 없는지 확인 후 핸드오프

#### Evaluator (검증자)
- **Playwright MCP**로 실제 쇼핑몰 접속, 익스텐션 동작 시나리오 실행
- 채점 기준:

| 기준 | 설명 | 가중치 |
|------|------|--------|
| **범용성** | 3개+ 쇼핑몰에서 80% 이상 필드 추출 성공 | 높음 |
| **기능성** | 추출→미리보기→전송 전체 플로우 완결 | 높음 |
| **UX** | 비개발자가 설정 화면 혼자 완결 가능 | 중간 |
| **데이터 정확성** | 가격/이미지 URL 오추출 없음 | 높음 |
| **안정성** | 팝업 크래시, 전송 실패 시 적절한 에러 표시 | 중간 |

---

## 스프린트 계약 예시 (이 프로젝트)

```markdown
## 스프린트 계약 #2 — DOM 파싱 + 팝업 미리보기

### 이번 스프린트 목표
- [ ] Content Script: 상품명·가격·대표 이미지 자동 추출 (3개 쇼핑몰 대상)
- [ ] Popup UI: 추출 결과 미리보기 화면 (수정 불가, 조회만)

### 이번 스프린트에서 하지 않을 것
- 리뷰 데이터 수집 → Sprint 4로 미룸
- 외부 API 실제 전송 → Sprint 3에서 구현
- 옵션/스펙 추출 → Sprint 3으로 미룸

### 평가 기준
- 쿠팡 상품 페이지: 상품명·가격·이미지 URL 3개 필드 모두 정확히 추출
- 네이버쇼핑 상품 페이지: 동일 3개 필드 추출 (구조 다름 → 폴백 로직 확인)
- 무신사 상품 페이지: 동일 3개 필드 추출
- 팝업: 추출 결과가 시각적으로 올바르게 표시됨

### 통과 조건
- 3개 쇼핑몰 모두 테스트 통과
- 팝업 크래시 없음
- 채점 범용성 80점 이상
```

## 에이전트 간 통신 — 파일 구조

```
chrome-extension/
├── .harness/
│   ├── spec.md                  # Planner → Generator (전체 제품 스펙)
│   ├── sprint-contract.md       # Generator ↔ Evaluator (스프린트 계약)
│   ├── sprint-result.md         # Generator → Evaluator (구현 결과 + 테스트 방법)
│   ├── evaluation.md            # Evaluator → Generator (쇼핑몰별 추출 결과, 점수)
│   ├── handoff.md               # 세션 간 컨텍스트 (파싱 성공/실패 쇼핑몰 목록 포함)
│   ├── design-system/
│   │   └── design-guide.md      # 팝업 UI 디자인 기준
│   └── templates/
│       └── 01-core-principles.md
├── src/
│   ├── content/                 # Content Script (DOM 파싱)
│   ├── popup/                   # React 팝업 UI
│   ├── background/              # Service Worker (MV3)
│   └── shared/                  # 타입, 유틸, 스키마
└── manifest.json
```

### 통신 규칙 (이 프로젝트 추가 규칙)
1. `evaluation.md`에는 반드시 **쇼핑몰명 + 어떤 필드가 실패했는지** 명시
2. `sprint-result.md`에는 테스트한 쇼핑몰 URL과 재현 방법 포함
3. 파싱 로직 변경 시 `handoff.md`의 "검증된 쇼핑몰 목록" 갱신 필수

## 이 프로젝트 적용 흐름

```
Sprint 0: 셋업
  → Planner가 전체 스펙 작성 (spec.md)
  → 기술 스택 확정 (Chrome MV3, TypeScript, React + Vite)

Sprint 1~2: DOM 파싱 MVP
  → Generator: Content Script + 기본 팝업
  → Evaluator: 3개 쇼핑몰 파싱 정확도 검증 (Playwright)
  → 반복 (5~8회)

Sprint 3~4: 전송 + 고급 추출
  → Generator: API 전송, 옵션/리뷰 추출
  → Evaluator: 전송 성공 여부 + 데이터 스키마 정합성 검증

Sprint 5~6: 설정 UI + 안정화
  → Evaluator: 비개발자 사용자 시나리오로 설정 화면 검증
  → 커버리지 확대 (10개+ 쇼핑몰)
```

## Skills / Hooks / Subagents 판단 (이 프로젝트)

| 상황 | 선택 | 이유 |
|------|------|------|
| 매 빌드 후 `manifest.json` 유효성 검사 | **PostToolUse Hook** | 매번 결정적으로 실행 필요 |
| MV3 금지 API 사용 차단 (`chrome.tabs.executeScript` 등) | **PreToolUse Hook** | 실행 전 반드시 차단 |
| 새 쇼핑몰 DOM 구조 분석 | **Explore Subagent** | 메인 컨텍스트 오염 없이 탐색 |
| 파싱 로직 위험한 리팩토링 | **Worktree Subagent** | 실패 시 변경 폐기 가능 |
| Chrome Extension 보안 리뷰 | **Skill** | 도메인 전문성, 자동 활성화 |

> **핵심 원칙**: 쇼핑몰이 추가될 때마다 Evaluator가 검증 — 이것만큼은 Hook이 아니라 **스프린트 계약의 통과 조건**으로 강제한다.
