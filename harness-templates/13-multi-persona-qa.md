# 13. 멀티 페르소나 QA 프로토콜 — 쇼핑몰 데이터 수집 크롬 익스텐션

> 이 익스텐션의 사용자는 비개발자다. 개발자 관점에서 "동작한다"고 해도, 비개발자가 목표를 완수하지 못하면 실패다.
> 3개 에이전트가 병렬로 검수하고, 사용자 관점이 최종 타이브레이커다.

---

## 이 프로젝트에서의 QA 초점 매핑

| 페르소나 | 이 프로젝트에서의 핵심 관심사 |
|---------|---------------------------|
| **디자이너** | 팝업 UI 일관성, 상태 전환 명확성, 필드 선택/설정 화면의 구조 |
| **개발자** | DOM 파싱 견고성, Chrome API 올바른 사용, 보안(토큰 저장), MV3 규격 준수 |
| **사용자** | 클릭 한 번으로 수집 완료되는가, 설정 없이도 첫 시작 가능한가, 오류 시 무엇을 해야 하는지 아는가 |

## 하네스 레벨 적용 가이드

| 스프린트 | 권장 레벨 | 이유 |
|---------|---------|------|
| Sprint 0–1 (셋업/팝업 뼈대) | Level 0 | 프로토타입, 구조 실험 단계 |
| Sprint 2–3 (DOM 파싱/전송) | Level 1 | 핵심 기능, 단일 Evaluator로 충분 |
| Sprint 4–5 (설정 UI/커버리지) | Level 2 | 비개발자 UX가 핵심, 3-페르소나 필요 |
| Sprint 6 (안정화/릴리즈) | Level 2 (Strict) | 외부 사용자 대상 완성품 |

## 페르소나 A: 디자이너 체크리스트 — 익스텐션 팝업

```markdown
## 디자이너 검수 — 쇼핑몰 데이터 수집 익스텐션

### 팝업 UI 일관성
- [ ] 팝업 전체가 design-guide.md 색상/폰트/간격 토큰을 준수하는가
- [ ] 하드코딩된 색상 값(#ffffff, px 단위 직접 기입 등)이 없는가
- [ ] 팝업 너비(보통 360–400px)에서 레이아웃이 깨지지 않는가
- [ ] 스크롤 없이 핵심 액션(수집 버튼)에 접근 가능한가

### 상태 표시 명확성
- [ ] 수집 중 / 수집 완료 / 수집 실패 / 미설정 상태가 시각적으로 구분되는가
- [ ] 로딩 스피너 또는 프로그레스가 존재하는가 (전송 중 무반응 금지)
- [ ] 성공 피드백(토스트 또는 배지)이 명확한가
- [ ] 에러 상태에서 아이콘/색상이 경고를 전달하는가

### 설정 화면 구조
- [ ] 엔드포인트 등록 / 인증 토큰 / 전송 포맷 설정이 논리적 순서로 배치되는가
- [ ] 필수 설정과 선택 설정이 시각적으로 구분되는가
- [ ] 수집 필드 선택 UI에서 선택된 항목이 명확히 하이라이트되는가
- [ ] "저장 완료" 피드백이 존재하는가

### 미리보기 화면
- [ ] 추출된 데이터가 가독성 있는 형식으로 표시되는가
- [ ] 긴 텍스트(상품 설명, 리뷰)가 잘리지 않거나 적절히 축약되는가
- [ ] 이미지 미리보기가 있다면 비율이 왜곡되지 않는가

### UX 라이팅
- [ ] 버튼 라벨이 행동을 명확히 설명하는가 ("저장" vs "전송" vs "수집" — 혼용 금지)
- [ ] 에러 메시지가 원인과 해결 방법을 함께 안내하는가
- [ ] 비개발자 용어 사용 (API 엔드포인트 → "전송 주소", Auth Token → "인증 키" 등)
```

## 페르소나 B: 개발자 체크리스트 — 익스텐션 코드

```markdown
## 개발자 검수 — 쇼핑몰 데이터 수집 익스텐션

### Chrome MV3 규격 준수
- [ ] manifest.json의 manifest_version이 3인가
- [ ] background script가 Service Worker로 구현되었는가 (persistent: false)
- [ ] Content Security Policy 위반 없는가
- [ ] 필요한 permission만 선언되었는가 (최소 권한 원칙)
- [ ] host_permissions가 <all_urls>인 경우 이유가 정당한가

### DOM 파싱 견고성
- [ ] 상품 페이지가 아닌 페이지(목록, 메인, 결제)에서 안전하게 실패하는가
- [ ] SPA(React/Vue) 방식 쇼핑몰에서 동적 렌더링 후 파싱이 동작하는가
- [ ] null/undefined DOM 요소에 대한 방어 처리가 있는가
- [ ] 옵션이 없는 상품, 리뷰가 없는 상품 등 빈 필드 처리가 되는가
- [ ] iframe 내부 콘텐츠 파싱 시도 시 보안 오류를 적절히 처리하는가

### 인증 토큰 보안
- [ ] 토큰이 chrome.storage.local에 저장되는가 (localStorage 금지)
- [ ] 토큰이 console.log나 에러 응답에 노출되지 않는가
- [ ] 팝업 → Content Script → Background 메시지 전달 시 토큰이 평문으로 노출되지 않는가
- [ ] 토큰 삭제/갱신 기능이 있는가

### 메시지 패싱 & 비동기 처리
- [ ] Content Script ↔ Background ↔ Popup 메시지 채널이 명확히 정의되었는가
- [ ] chrome.runtime.sendMessage 응답이 항상 처리되는가 (미처리 Promise 없음)
- [ ] 페이지 이동/탭 전환 중 수집 요청 시 경쟁 조건이 없는가
- [ ] API 전송 실패 시 재시도 로직 또는 명확한 실패 처리가 있는가

### TypeScript & 코드 품질
- [ ] any 타입 남용이 없는가 (DOM 요소 타입 assertion 포함)
- [ ] 추출된 데이터 스키마가 타입으로 정의되어 있는가
- [ ] 빌드가 경고 없이 성공하는가
- [ ] 기존 테스트가 모두 통과하는가

### 성능
- [ ] DOM 파싱이 페이지 렌더링을 블로킹하지 않는가
- [ ] MutationObserver 사용 시 disconnect()가 적절히 호출되는가
- [ ] 대용량 이미지 URL 배열 등 데이터 페이로드에 사이즈 제한이 있는가
```

## 페르소나 C: 사용자 체크리스트 — 비개발자 데이터 운영자

> 당신은 코딩을 모르는 MD(머천다이저) 또는 가격 모니터링 담당자다.
> Chrome 익스텐션 설치 경험은 있지만, API나 JSON은 낯설다.

```markdown
## 사용자 검수 — 쇼핑몰 데이터 수집 익스텐션

### 첫 설치 후 경험 (온보딩)
- [ ] 설치 직후 무엇을 해야 하는지 안내가 있는가
- [ ] 엔드포인트와 인증 토큰을 어떻게 얻는지 설명이 있는가 (또는 링크)
- [ ] 설정 없이 아이콘을 클릭했을 때 "설정이 필요합니다"가 명확히 안내되는가
- [ ] 설정 완료까지 몇 단계인지 알 수 있는가

### 핵심 플로우: 상품 페이지 → 수집 → 전송
- [ ] 상품 페이지에서 아이콘 클릭 → 수집 시작이 즉시(3초 이내) 반응하는가
- [ ] 수집 중임을 알리는 피드백이 있는가 (스피너, 진행 표시)
- [ ] 수집 완료 후 무엇이 추출되었는지 미리볼 수 있는가
- [ ] 전송 버튼 클릭 → 성공 확인까지 흐름이 끊기지 않는가
- [ ] 전송 성공 후 "다음 상품으로 이동"이 자연스러운가

### 설정 화면
- [ ] 인증 토큰 입력 필드에 올바른 입력 형식 안내가 있는가
- [ ] 잘못된 토큰을 저장했을 때 즉시 경고가 있는가 (전송 실패 시점이 아닌)
- [ ] 수집 필드를 선택/해제하는 UI가 직관적인가
- [ ] 설정 저장 후 "저장됨" 확인이 있는가

### 오류 상황 대처
- [ ] 상품 페이지가 아닌 곳에서 클릭했을 때 알기 쉬운 안내가 나오는가
- [ ] 인터넷 연결 없을 때 "전송 실패 — 네트워크를 확인해주세요"처럼 구체적인가
- [ ] 수집 결과가 비어있을 때(파싱 실패) 무엇을 해야 하는지 안내가 있는가
- [ ] 오류 메시지에 "오류 코드 404" 같은 개발자 용어만 노출되지 않는가

### 반복 사용 효율
- [ ] 동일 사이트에서 두 번째 수집이 첫 번째보다 빠르고 설정 재입력 없이 가능한가
- [ ] 자동 전송 설정 시 클릭 한 번으로 수집+전송이 완결되는가
- [ ] 오늘 수집한 항목 수 / 마지막 전송 시각 등 현황을 확인할 수 있는가
```

## 이 프로젝트 특유의 토론 규칙

**충돌 시나리오 예시:**

| 충돌 상황 | 디자이너 의견 | 개발자 의견 | 타이브레이커 |
|----------|-------------|-----------|-----------|
| 토큰 입력 필드에 실시간 유효성 검사 추가 | "즉각 피드백 필요" | "추가 API 호출 비용" | 사용자: "잘못 저장 후 전송 실패가 더 혼란스럽다" → 디자이너 채택 |
| 파싱 실패 시 raw HTML 표시 | "비전문가 화면에 부적합" | "디버깅에 유용" | 사용자: "의미 없는 코드 보임" → 디자이너 채택, 로그는 별도 개발자 모드 |
| 수집 필드 전체 선택/해제 버튼 | "공간 차지" | "구현 단순" | 사용자: "100개 필드 개별 클릭은 불가능" → 사용자가 필요 → 추가 |

**이 프로젝트 고정 우선순위:**
```
1순위: 사용자 (비개발자 운영자가 목표 완수 가능한가)
2순위: 개발자 (보안 — 특히 토큰 노출, MV3 규격)
3순위: 디자이너 (팝업 UI 일관성)

예외: 토큰/인증 관련 보안 이슈는 항상 최우선 (관점 무관)
```

## PASS/FAIL 기준 (이 프로젝트)

### 자동 FAIL 조건 (심각도 무관)
- 인증 토큰이 chrome.storage.local 이외의 저장소에 저장되는 경우
- 상품 페이지가 아닌 곳에서 아이콘 클릭 시 익스텐션이 크래시되는 경우
- 전송 실패 시 사용자에게 아무 피드백이 없는 경우
- MV3 Service Worker 규격 위반으로 Chrome Web Store 게시 불가한 경우

### 권고 사항 처리 (다음 스프린트 이월 가능)
- 특정 쇼핑몰에서 파싱 정확도 개선
- 수집 이력 로그 UI
- 다국어 지원

## 비용 판단 기준

- **Sprint 0–3**: Level 0/1 — DOM 파싱 정확도, API 연동 등 기능 검증이 우선. 3-페르소나 불필요
- **Sprint 4+**: Level 2 — 비개발자 UX가 제품 가치의 핵심. 3-페르소나 필수
- **FAIL → 재수정 루프**: 최대 3회. 3회 초과 시 "범용 파서 접근법 재검토" 또는 특정 쇼핑몰 예외 처리 방식 논의
