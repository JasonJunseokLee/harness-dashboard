# 08. 하네스 적용 판단 체크리스트
### 쇼핑몰 데이터 수집 크롬 익스텐션 맞춤 버전

---

## 빠른 판단 플로우차트

```
작업 요청 수신
    │
    ├── 버그 수정 / 단순 설정 변경 / 문서 수정?
    │   └── 예 → 하네스 불필요 (Level 0)
    │
    ├── DOM 파싱 로직 / 신규 쇼핑몰 커버리지 / UI 컴포넌트 단건?
    │   └── 예 → Evaluator 추가 고려 (Level 1)
    │
    ├── 범용 파싱 알고리즘 설계 / 전송 포맷 아키텍처 / 스프린트 전체?
    │   └── 예 → Planner 추가 (Level 2)
    │
    └── 비개발자 비기획자도 QA 기준이 필요한가?
        └── 예 → 채점 기준 명시 후 Evaluator 운영 (Level 1~2)
```

## Evaluator 적용 판단

### 적용해야 할 때

| 상황 | 이 프로젝트에서의 의미 |
|------|------|
| 품질 임계값이 타협 불가 | 잘못 추출된 가격·옵션 데이터가 플랫폼 전체에 오염됨 |
| 주관적 판단 튜닝 필요 | "범용 파싱"이 맞는지 기준이 불명확 → 채점 루브릭 필요 |
| 반복 개선이 유의미한 차이를 만듦 | 쇼핑몰 DOM 구조는 다양해서 1회 생성으로 완성 어려움 |
| 비개발자가 결과를 검토해야 함 | 추출 미리보기 UI, 상태 표시 텍스트 등 UX 품질 판단 |

### 적용하지 않아도 될 때

| 상황 | 이 프로젝트에서의 예시 |
|------|------|
| 객관적 기준으로 자동 검증 가능 | Vitest 단위 테스트가 통과하면 충분한 파싱 유틸 함수 |
| 빠른 프로토타입이 목적 | Sprint 0 셋업 단계 — 환경 구성, 매니페스트 초안 |
| 기존 코드에 단순 기능 추가 | 엔드포인트 URL 저장 방식 수정, 스토리지 키 변경 등 |

## Planner 적용 판단

### 적용해야 할 때

| 상황 | 이 프로젝트에서의 예시 |
|------|------|
| 여러 기능이 상호 의존적 | DOM 파서 ↔ 필드 선택 ↔ 전송 포맷 ↔ 미리보기 — 동시에 설계 필요 |
| AI 기능 통합 기회가 있음 | 범용 파싱에 LLM 보조 가능 여부 판단 |
| 프로젝트 규모가 큼 (3+ 스프린트) | Sprint 0~6, 총 7개 스프린트 → 전체 로드맵 필요 |
| 스펙이 모호한 신규 컴포넌트 | "수집 필드 선택 UI"의 인터랙션 방식이 미정인 경우 |

### 적용하지 않아도 될 때

| 상황 | 이 프로젝트에서의 예시 |
|------|------|
| 요구사항이 이미 명확함 | 인증 토큰 저장: `chrome.storage.sync` 사용으로 확정된 경우 |
| 기존 아키텍처에 기능 추가 | 파서가 확정된 이후 새 쇼핑몰 셀렉터 추가 |
| 단일 기능 구현 | 팝업 UI에 "복사" 버튼 하나 추가 |

## 이 프로젝트의 스프린트별 권장 수준

| 스프린트 | 주요 작업 | 권장 레벨 | 이유 |
|----------|-----------|-----------|------|
| Sprint 0 | 환경 셋업, 기술 스택 확정 | **Level 0** | 셋업은 체크리스트 수준, 품질 판단 불필요 |
| Sprint 1 | 범용 DOM 파서 MVP | **Level 1** | 파싱 품질이 핵심 — Evaluator로 추출 정확도 검증 |
| Sprint 2 | 옵션/스펙/리뷰 추출 | **Level 1~2** | 데이터 구조 설계 복잡도에 따라 Planner 추가 |
| Sprint 3 | 팝업 UI + 미리보기 | **Level 1** | 비개발자 UX 품질 중요 — Evaluator로 사용성 평가 |
| Sprint 4 | API 전송 + 인증 | **Level 1** | 보안/안정성 품질 임계값 높음 |
| Sprint 5 | 쇼핑몰 커버리지 확대 | **Level 0~1** | 반복 작업 — 단순 셀렉터 추가는 Level 0 |
| Sprint 6 | 안정화 + 스토어 배포 | **Level 2** | 완성품 수준, 디자인+기능+코드 품질 모두 중요 |

## 이 프로젝트 특화 판단 기준

### DOM 파싱 작업
범용 파서는 이 프로젝트의 핵심 IP다. 단독 생성 후 바로 배포하지 말 것.

```
DOM 파서 구현 요청
    │
    ├── 특정 1개 쇼핑몰 셀렉터 추가 → Level 0 (테스트로 검증)
    │
    ├── 범용 파싱 알고리즘 신규 설계 → Level 1 (추출 정확도 Evaluator)
    │
    └── 파서 + 필드 매핑 + 미리보기 동시 설계 → Level 2
```

### 전송 포맷/API 연동 작업
잘못된 포맷이 고객사 플랫폼에 직접 영향을 미친다.

```
API 연동 작업 요청
    │
    ├── 기존 포맷에 필드 추가 → Level 0
    │
    ├── 신규 전송 포맷 스키마 설계 → Level 1 (호환성 Evaluator)
    │
    └── 멀티 엔드포인트 + 인증 + 재전송 로직 → Level 2
```

### 팝업 UI 작업
비개발자 타겟 — UX 품질 기준이 코드 기준보다 중요할 수 있다.

```
UI 작업 요청
    │
    ├── 버튼 하나, 텍스트 수정 → Level 0
    │
    ├── 수집 상태 표시 / 추출 미리보기 화면 → Level 1
    │    (채점 기준: 비개발자가 3초 안에 상태를 파악할 수 있는가)
    │
    └── 팝업 전체 UI 리디자인 → Level 2
```

## 점진적 확대 전략

```
1단계: Solo agent로 시작
    │
    ├── 파싱 결과가 만족스러움 → 완료
    │
    └── 결과가 부족함
        │
        ├── 추출 데이터 오류 / UX 품질 → Evaluator 추가 (Level 1)
        ├── 기능 간 연결 설계 오류 → Planner 추가 (Level 2)
        └── 둘 다 → 풀 하네스 (Level 2)
```

**이 프로젝트에서 특히 주의**: 범용 파서의 "정확도"는 주관적이다. 처음부터 채점 기준(`상품명 일치율`, `가격 포맷 정규화`, `이미지 URL 유효성`)을 Evaluator에 명시해야 루프가 의미 있다.

## 최종 의사결정 체크리스트

새 작업 시작 시 아래를 순서대로 확인:

- [ ] **작업 복잡도**: 단순(셀렉터 추가) / 중간(신규 추출 필드) / 복잡(파서 아키텍처)
- [ ] **품질 요구**: 프로토타입 / 내부 검증용 / 고객사 연동 완성품
- [ ] **단독 처리 가능성**: 높음(설정 변경) / 보통(단일 파서) / 낮음(범용 알고리즘)
- [ ] **스펙 명확도**: 명확(기존 구조 추가) / 보통 / 모호(새 쇼핑몰 유형)
- [ ] **데이터 오염 위험**: 없음 / 낮음 / 높음 (고객사 플랫폼 직접 영향)
- [ ] **비개발자 UX 판단 필요**: 없음 / 약간 / 필수
- [ ] **예산/속도**: 빠른 검증 우선 / 균형 / 품질 우선
- [ ] **결론**: Level 0 / Level 1 / Level 2

> **이 프로젝트의 기본값**: 범용 DOM 파서와 고객사 연동 관련 작업은 **Level 1 이상**을 기본으로 한다. 데이터 오염 위험이 크기 때문이다.
